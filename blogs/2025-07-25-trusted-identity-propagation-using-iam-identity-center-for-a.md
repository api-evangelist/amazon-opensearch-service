<p>Enterprise customers of <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/what-is.html" rel="noopener" target="_blank">Amazon OpenSearch Service</a> require comprehensive security controls with seamless authentication and authorization mechanisms when accessing data in provisioned domains and <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/serverless-overview.html" rel="noopener" target="_blank">Amazon OpenSearch Serverless</a> collections. Security teams within these organizations must not only maintain compliance with enterprise policies but also need to make sure that their users can access data securely, with robust identity management. <a href="https://docs.aws.amazon.com/singlesignon/latest/userguide/what-is.html" rel="noopener" target="_blank">AWS IAM Identity Center</a> is a popular mechanism for identity management that provides single sign-on (SSO) capabilities for these enterprise customers. IAM Identity Center can use <a href="https://wiki.oasis-open.org/security/Saml2TechOverview" rel="noopener" target="_blank">Security Assertion Markup Language</a> (SAML) with both <a href="https://aws.amazon.com/blogs/big-data/role-based-access-control-in-amazon-opensearch-service-via-saml-integration-with-aws-iam-identity-center/" rel="noopener" target="_blank">OpenSearch Service provisioned domains</a> and <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/serverless-saml.html" rel="noopener" target="_blank">OpenSearch Serverless</a>. Now, by using <a href="https://docs.aws.amazon.com/singlesignon/latest/userguide/trustedidentitypropagation-overview.html" rel="noopener" target="_blank">trusted identity propagation</a>, IAM Identity Center provides a new, direct method for accessing data in OpenSearch Service.</p> 
<p>In this post, we outline how you can take advantage of this new access method to simplify data access using the <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/application.html" rel="noopener" target="_blank">OpenSearch UI</a> and still maintain robust role-based access control for your OpenSearch data.</p> 
<h2>Trusted identity propagation overview</h2> 
<p>Trusted identity propagation in IAM Identity Center adds the identity context of a user to a role when accessing OpenSearch Service, which in turn uses this context to authorize and scope OpenSearch data access. This simplifies the authentication and authorization flow for customers because the applications access the data on their behalf. Users or user agents need not be present between the application and the backend services for this authorization to happen, unlike methods like SAML where a user agent needs to be present between these entities as a go-between for exchanging assertions. This flexibility helps simplify accessing a wide variety of data sources such as data residing within the <a href="https://aws.amazon.com/vpc/" rel="noopener" target="_blank">Amazon Virtual Private Cloud (Amazon VPC)</a> of an OpenSearch Service domain, or an OpenSearch Serverless collection. By using the OpenSearch UI, you can additionally simplify the backend connections, resulting in seamless access to the data. The following figures shows how the identity propagation works with OpenSearch Service.</p>

<h2>Prerequisites</h2> 
<p>Before starting to use IAM Identity Center with OpenSearch Service, there are a few options that you must enable. To start, set up an organization or account instance of IAM Identity Center following the instruction <a href="https://docs.aws.amazon.com/singlesignon/latest/userguide/identity-center-instances.html" rel="noopener" target="_blank">in this guide</a>. For OpenSearch Service-provisioned domains, you must enable the <strong>IAM Identity Center (IDC) Authentication –<em>new</em></strong> option. You can do this though <a href="https://aws.amazon.com/cloudformation" rel="noopener" target="_blank">AWS CloudFormation</a>, OpenSearch <a href="https://docs.aws.amazon.com/opensearch-service/latest/APIReference/API_Operations_Amazon_OpenSearch_Service.html" rel="noopener" target="_blank">REST API</a>, <a href="https://aws.amazon.com/developer/tools/" rel="noopener" target="_blank">AWS SDK</a>, or the AWS Management Console.</p> 
<p><strong>To enable Identity Center using the console</strong></p> 
<ol> 
 <li>To add the capability for an existing provisioned domain, go to the OpenSearch Service console and navigate to the <strong>Security configuration</strong> tab and choose <strong>Edit</strong>.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/20/BDB-5259-image-1.jpeg" rel="noopener" target="_blank"><img alt="Amazon OpenSearch Service console highlighting security settings, cluster health status, and VPC endpoint configuration" class="alignnone size-full wp-image-79541" height="476" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/20/BDB-5259-image-1.jpeg" width="1431" /></a></p> 
<ol start="2"> 
 <li>After this step, or if you are creating new domain, select the check box for <strong>IAM Identity Center (IDC) Authentication – <em>new</em></strong>.</li> 
 <li>You have various options to choose for <strong>Subject Key</strong> and <strong>Roles Key </strong>depending upon how you want to establish your role-based access control discussed later in this post. For now, select <strong>UserName</strong> for <strong>Subject Key</strong> and <strong>GroupName</strong> for <strong>Roles Key</strong>.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-2.jpeg" rel="noopener" target="_blank"><img alt="AWS IAM Identity Center SSO configuration interface with API authentication controls and identity mapping settings" class="alignnone size-full wp-image-79778" height="332" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-2.jpeg" width="940" /></a></p> 
<ol start="4"> 
 <li>For OpenSearch Serverless, choose <strong>Serverless </strong>in the navigation pane, then <strong>Security</strong> and <strong>Authentication</strong>. Choose <strong>Edit</strong> in the <strong>IAM Identity Center (IdC) authentication – <em>new</em></strong> section.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-3.jpeg" rel="noopener" target="_blank"><img alt="AWS OpenSearch Service authentication management console with SAML providers list and IAM Identity Center integration status" class="alignnone size-full wp-image-79779" height="492" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-3.jpeg" width="1429" /></a></p> 
<ol start="5"> 
 <li>Select the checkbox for <strong>Authenticate with IAM Identity Center</strong>, and then choose <strong>Save</strong>.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-4.jpeg" rel="noopener" target="_blank"><img alt="AWS IAM Identity Center configuration modal for enabling centralized authentication and resource access management" class="alignnone size-full wp-image-79780" height="252" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-4.jpeg" width="1432" /></a></p> 
<ol start="6"> 
 <li>Select the checkbox for <strong>Authentication with IAM Identity Center</strong> under <strong>Single sign-on authentication</strong>, when creating an OpenSearch UI application. For step-by-step instructions on how to create an OpenSearch UI application, see <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/application-getting-started.html#create-application" rel="noopener noreferrer" target="_blank">Creating an OpenSearch UI application</a></li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-5.jpeg" rel="noopener" target="_blank"><img alt="AWS single sign-on setup interface with option to enable IAM Identity Center for unified access management" class="alignnone size-full wp-image-79782" height="213" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-5.jpeg" width="924" /></a></p> 
<p>After these steps, you’re ready to configure IAM Identity Center by creating new users and groups, or by using existing user identities.</p> 
<h2>Propagating IAM Identity Center identities</h2> 
<p>Currently, adding single sign-on authentication with IAM Identity Center can be done while setting up a new <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/application-getting-started.html#create-application" rel="noopener noreferrer" target="_blank">OpenSearch UI application</a>. Use the following steps to create a <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/application-getting-started.html#create-application" rel="noopener noreferrer" target="_blank">new OpenSearch UI application</a>. Note that single sign-on currently cannot be turned on after an application is created. After single sign-on is enabled, you should see an <strong>AWS managed application</strong> under <strong>Applications</strong> in the Identity Center console.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-6.jpeg" rel="noopener" target="_blank"><img alt="AWS Identity Center interface showing managed applications tab with OpenSearch integration and service details" class="alignnone size-full wp-image-79784" height="260" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-6.jpeg" width="1428" /></a></p> 
<h3>Assigning users and groups</h3> 
<p>After the application is created and the status shows as <strong>Active</strong>, you need to assign users and groups to the application. This assignment is important and recommended because these assignments determine the scope and permissions for data access within OpenSearch Service. To do this, select the application you created in the previous steps in the OpenSearch Service console. Here, you will see an option for <strong>IAM Identity Center user and groups</strong> under <strong>Single sign-on authentication</strong>. Choose <strong>Assign users and groups</strong> and select the appropriate Identity Center users and groups.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-7.jpeg" rel="noopener" target="_blank"><img alt=" AWS IAM Identity Center dashboard showing connected status, role details, and option to assign users and groups for SSO" class="alignnone size-full wp-image-79785" height="303" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-7.jpeg" width="1429" /></a></p> 
<p>For OpenSearch Serverless, you must create a new <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/serverless-data-access.html" rel="noopener" target="_blank">data</a> access policy or add a rule to an existing one to grant IAM Identity Center principals appropriate permissions to access the collections. For example, the following figure shows a data access policy that grants specific permissions to a one user with <strong>Rule 1</strong> and provide a more restrictive permission to a group with <strong>Rule 2</strong>.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-8.jpeg" rel="noopener" target="_blank"><img alt="Amazon OpenSearch Service console displaying sample data access policy with IAM principal and collection/index permissions" class="alignnone size-full wp-image-79786" height="576" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-8.jpeg" width="1428" /></a></p> 
<p>At this point your OpenSearch Service domains, OpenSearch Serverless collections and OpenSearch UI are set up for identity propagation.</p> 
<h3>Fine grained access control for IAM Identity Center identities</h3> 
<p><a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/fgac.html" rel="noopener" target="_blank">Fine grained access control</a> is a role-based access control for OpenSearch Service that provides security at index, document, and field levels for provisioned domains. You can choose what aspects of identity context you propagate to OpenSearch Service. You can choose between <em>UserId</em>, <em>UserName</em>, and Email for your Subject keys, and <em>GroupId</em> and <em>GroupName</em> for your Roles key. This configuration is important because the values of the properties in the identity context are used to match exactly with the <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/fgac.html#fgac-concepts" rel="noopener" target="_blank">user and backend role mapping</a> within OpenSearch Service provisioned domains. Note that if IAM Identity Center sign-on isn’t enabled, OpenSearch Service can only evaluate the <a href="https://docs.aws.amazon.com/AmazonS3/latest/API/sig-v4-authenticating-requests.html" rel="noopener" target="_blank">request signature with AWS signature Version 4</a>. This means that the role your OpenSearch UI will use won’t contain identity context for authorization. To complete authorization, add the values of the identity context fields to the OpenSearch role mapping. See <em>Mapping roles to users</em> under <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/fgac.html#fgac-access-control" rel="noopener" target="_blank">Managing permissions</a>. Role mapping can be done using OpenSearch REST API, AWS SDK, or using OpenSearch Dashboards.</p> 
<p><strong>To map roles using OpenSearch Dashboards</strong></p> 
<ol> 
 <li>From the menu icon <a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-9.png" rel="noopener" target="_blank"><img alt="" class="alignnone size-full wp-image-79787" height="29" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-9.png" width="39" /></a>&nbsp;on the top left corner or your screen, select <strong>Management</strong>, <strong>Security</strong>, <strong>Roles</strong>, <em>&lt;Your role&gt;</em>.</li> 
 <li>Choose the <strong>Mapped Users</strong> tab and select <strong>Manage mapping</strong>.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-10.jpeg" rel="noopener" target="_blank"><img alt="AWS security console showing Map User interface with internal user creation and backend role management for IAM" class="alignnone size-full wp-image-79788" height="464" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-10.jpeg" width="1431" /></a></p> 
<ol start="3"> 
 <li>When mapping the role, make sure that you enter the values corresponding to the Subject key. This value must be the same as in your identity context. Additionally, use the Roles key to assign access-based IAM Identity Center groups.</li> 
</ol> 
<p>With OpenSearch Serverless, the granularity of access control is at the index level so you will need to add additional rules in the Data Access Policy to control principals who can access collections or indices within a collection.</p> 
<h3>Verifying identity propagation</h3> 
<p>The final step is to verify identity propagation.</p> 
<ol> 
 <li>Open the OpenSearch UI application and select <strong>IAM Identity Center</strong> from the <strong>Login</strong> drop down.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-11-1.jpeg" rel="noopener" target="_blank"><img alt="" class="alignnone wp-image-79797 size-full" height="419" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-11-1.jpeg" width="876" /></a></p> 
<ol start="2"> 
 <li>After you complete the login process with IAM Identity Center, the OpenSearch UI will open. Choose the user icon in the lower left corner of the screen to verify that it’s your correct principal from Identity Center. It should match the Identity Center property you chose earlier.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-12.jpeg" rel="noopener" target="_blank"><img alt="OpenSearch welcome interface showing available workspaces, documentation links, and user profile menu with logout option" class="alignnone size-full wp-image-79791" height="786" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-12.jpeg" width="659" /></a></p> 
<ol start="3"> 
 <li>To verify correct identity propagation, choose the <strong>Dev tools</strong> <a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-13.png" rel="noopener" target="_blank"><img alt="" class="alignnone size-full wp-image-79792" height="26" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/24/BDB-5259-image-13.png" width="33" /></a>&nbsp;icon just above the user profile icon in the bottom left corner of the screen.</li> 
 <li>Select the correct OpenSearch domain or OpenSearch Serverless collection data source in the top right corner of the screen and run a <code>_search</code> query. You should see results from the data source confirming that the identity is correctly propagated to OpenSearch Service.</li> 
</ol>

<h2>Conclusion</h2> 
<p>In this post, we showed you how to use <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/idc-aos.html" rel="noopener" target="_blank">Trusted Identity Propagation using IAM Identity Center for Amazon OpenSearch Service</a>, providing a streamlined approach to secure data access while maintaining robust access controls. This solution offers several key benefits:</p> 
<ul> 
 <li><strong>Simplified authentication</strong>: By eliminating the need for user agents between applications and backend services, the solution streamlines the authentication process compared to traditional SAML-based approaches.</li> 
 <li><strong>Enhanced security</strong>: The integration maintains comprehensive security controls while providing seamless authentication and authorization mechanisms for both OpenSearch Service provisioned domains and Amazon OpenSearch Serverless collections.</li> 
 <li><strong>Flexible identity management</strong>: Organizations can use existing IAM Identity Center implementations to manage user access, making it easier to maintain compliance with enterprise security policies.</li> 
 <li><strong>Fine-grained access control</strong>: The solution supports detailed access control at the index, document, and field level for provisioned domains, allowing organizations to implement precise security measures.</li> 
</ul>
