<p>Organizations today face the challenge of managing and deriving insights from an ever-expanding universe of data in real time. Industrial <a href="https://aws.amazon.com/what-is/iot/" rel="noopener noreferrer" target="_blank">Internet of Things</a> (IoT) sensors stream millions of temperature, pressure, and performance metrics from field equipment every second. Ecommerce platforms need to surface relevant products from vast catalogs instantly. Security teams must analyze system logs in real time to detect threats. As data volumes grow, organizations increasingly struggle with fragmented monitoring tools that create critical visibility gaps and slow incident response times. The cost of commercial observability solutions becomes prohibitive, forcing teams to manage multiple separate tools and increasing both operational overhead and troubleshooting complexity. Across these diverse scenarios, the ability to efficiently search, analyze, and visualize data in real time has become crucial for business success.</p> 
<p><a href="https://aws.amazon.com/opensearch-service/" rel="noopener noreferrer" target="_blank">Amazon OpenSearch Service</a> addresses these challenges by providing a fully managed search and analytics service. This managed service configures, manages, and scales OpenSearch clusters so you can focus on your search workloads and end customers. <a href="https://aws.amazon.com/opensearch-service/features/serverless/" rel="noopener noreferrer" target="_blank">Amazon OpenSearch Serverless</a> further makes it straightforward to run search and log analytics workloads by automatically scaling compute and storage resources up and down to match your application’s demands—with no infrastructure to manage. Whether you’re processing continuous streams of IoT telemetry, enabling product discovery, or performing security analytics, OpenSearch Service scales to meet your needs.</p> 
<p>In this post, we walk you through a search application building process using Amazon OpenSearch Service. Whether you’re a developer new to search or looking to understand OpenSearch fundamentals, this hands-on post shows you how to build a search application from scratch—starting with the initial setup; diving into core components such as indexing, querying, result presentation; and culminating in the execution of your first search query.</p> 
<h2>Components of OpenSearch Service</h2> 
<p>Before building your <a href="https://github.com/aws-samples/sample-for-amazon-opensearch-tutorials-101" rel="noopener noreferrer" target="_blank">first search application</a>, it’s important to understand some key architectural components in OpenSearch. The fundamental unit of information in OpenSearch is a <a href="https://docs.opensearch.org/docs/latest/getting-started/intro/#document" rel="noopener noreferrer" target="_blank">document</a> stored in JSON format. These documents are organized into <a href="https://docs.opensearch.org/docs/latest/getting-started/intro/&quot; \l &quot;index" rel="noopener noreferrer" target="_blank">indices</a>—collections of related documents that function similar to database tables. When you search for information, OpenSearch queries these indices to find matching documents.</p> 
<p>OpenSearch operates on a distributed architecture where multiple servers, called nodes, work together in a cluster or domain. Each cluster can utilize <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/managedomains-dedicatedmasternodes.html" rel="noopener noreferrer" target="_blank">dedicated master nodes</a> that focus solely on cluster management tasks, such as maintaining cluster state, managing indices, and orchestrating shard allocation. These specialized nodes enhance cluster stability by offloading cluster management duties from data nodes. Data nodes, on the other hand, handle the storage, indexing, and querying of data—essentially performing the heavy lifting of data operations. Together, they provide scalability, availability, and efficient data processing in the cluster. Configure <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/Dedicated-coordinator-nodes.html" rel="noopener noreferrer" target="_blank">dedicated coordinator nodes</a> that specialize in routing and distributing search and indexing requests across the cluster. These nodes reduce the load on data nodes, which allows them to focus on data storage, indexing, and search operations.</p> 
<p>Coordinator nodes in OpenSearch are most beneficial in the following scenarios:</p> 
<ol> 
 <li><strong>Large cluster deployments</strong> – When managing substantial data volumes across many nodes.</li> 
 <li><strong>Query-intensive workloads</strong> – For environments handling frequent search queries or aggregations, especially those with complex date histograms or multiple aggregations, benefit from faster query processing.</li> 
 <li><strong>Heavy dashboard utilization</strong> – <a href="https://opensearch.org/docs/latest/dashboards/" rel="noopener noreferrer" target="_blank">OpenSearch Dashboards</a> can be resource-intensive. Offloading this responsibility to dedicated coordinator nodes reduces the strain on data nodes.</li> 
</ol> 
<p>To manage large datasets efficiently, OpenSearch splits indices into smaller pieces called shards. Each shard is <a href="https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html&quot; \l &quot;bp-sharding-strategy" rel="noopener noreferrer" target="_blank">distributed across the cluster</a>, with a recommended size of 10–50 GB for optimal performance. For reliability and high availability, OpenSearch maintains replica copies of these shards on different nodes, which means that your data remains accessible even if some nodes fail.</p> 
<p>Search operations in OpenSearch are powered by inverted indices, a data structure that maps terms to the documents containing them. The <a href="https://opensearch.org/docs/latest/search-plugins/keyword-search/" rel="noopener noreferrer" target="_blank">BM25 ranking algorithm</a> helps make sure that search results are relevant to users’ queries. Although searches happen in near real time, with configurable refresh intervals, individual document retrievals are immediate.</p> 
<p>This architecture provides the foundation for handling high-volume IoT data streams, complex full-text search operations, and real-time analytics, all while maintaining fault tolerance. Understanding these components will help you make informed decisions as you build your search application.OpenSearch Dashboards is a visualization and analytics tool for exploring, analyzing, and visualizing data in real time. It provides an intuitive interface for querying, monitoring, and reporting on OpenSearch data using visualizations such as charts, graphs, and maps. Key features include interactive dashboards, alerting, anomaly detection, security monitoring, and trace analytics.</p> 
<h2>Sample Amazon OpenSearch Service tutorial application overview</h2> 
<p>The following architecture diagram demonstrates how to build and deploy a scalable, fully managed search application on <a href="https://aws.amazon.com/" rel="noopener noreferrer" target="_blank">Amazon Web Services</a> (AWS). The architecture uses Amazon OpenSearch Service for indexing and searching data. The UI application is deployed on <a href="https://aws.amazon.com/apprunner/" rel="noopener noreferrer" target="_blank">AWS App Runner</a> and interacts with Amazon OpenSearch Service through secure serverless <a href="https://aws.amazon.com/api-gateway/" rel="noopener noreferrer" target="_blank">Amazon API Gateway</a> and <a href="https://aws.amazon.com/lambda" rel="noopener noreferrer" target="_blank">AWS Lambda</a>.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/08/BDB-5102-image-1.png"><img alt="Scope of Solution" class="alignnone size-full wp-image-79053" height="322" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/08/BDB-5102-image-1.png" style="margin: 10px 0px 10px 0px;" width="761" /></a></p> 
<p>Here is the end-to-end workflow for our application detailing how user requests are handled from initial access through to data retrieval or indexing:</p> 
<ol> 
 <li>Users access the application through AWS App Runner, which hosts the frontend interface.</li> 
 <li><a href="https://aws.amazon.com/cognito/" rel="noopener noreferrer" target="_blank">Amazon Cognito</a> handles user authentication and authorization for secure access to the application.</li> 
 <li>When users interact with the application, their requests are sent to API Gateway. API Gateway communicates with Amazon Cognito to verify user authentication status. It serves as the primary entry point for all API operations and routes the requests appropriately. It forwards requests to Lambda functions within the virtual private cloud (VPC).</li> 
 <li>Lambda functions process the requests, performing either:</li> 
 <li>Data indexing operations into OpenSearch Service</li> 
 <li>Search queries against the OpenSearch Service cluster</li> 
 <li>The OpenSearch Service cluster resides within a private subnet in a VPC for enhanced security.</li> 
</ol>

<h2>Keyword searches</h2> 
<p>OpenSearch Service offers multiple search features. For an exhaustive list, refer to <a href="https://opensearch.org/docs/latest/search-plugins/" rel="noopener noreferrer" target="_blank">Search features</a>. We focus on a few keyword search types to help you get started with OpenSearch.</p> 
<p>With the product catalog in OpenSearch, you can perform prefix searches through the search application’s intuitive interface. To better understand the search functionality, expand the <strong>Guide section </strong>at the top of the interface. This interactive guide explains how various kinds of searches work, complete with a practical example in context of the product catalog dataset. The guide includes best practices and a link to the detailed documentation to help you make the most of OpenSearch’s powerful query capabilities.</p> 
<p>You can do a prefix search on any of the three key search fields: <strong>Title</strong>, <strong>Description</strong>, or <strong>Color</strong>.</p> 
<p>A typical prefix match query looks like this:</p> 
<div class="hide-language"> 
 <pre><code class="lang-json">{
  "query": {
    "match_phrase_prefix": {
      "attribute_name": {
        "query": "attribute_value",
        "max_expansions": 10,
        "slop": 1
      }
    }
  }
}</code></pre> 
</div> 
<p>You can use this query pattern to find documents where specific fields begin with your search term, offering an intuitive “starts with” search experience.</p> 
<p>The following image illustrates a practical example of the <strong>Prefix Match </strong>search. Entering “Ru” in the title field matches products with titles such as “Running”, “Runners” and “Ruby.” Prefix Match search is particularly useful when users only remember the beginning of a product name or are searching across multiple variations or simply exploring product categories.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/18/bdb-5102-u-image-3.png"><img alt="Prefix Match example" class="alignnone wp-image-79440 size-full" height="889" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/18/bdb-5102-u-image-3.png" style="margin: 10px 0px 10px 0px;" width="1430" /></a></p> 
<p><strong>Multi Match</strong> search enables searching across multiple fields simultaneously. For example, you can search for “Coral” across product title, description, and color fields simultaneously. The search query can be customized using field boosting in which matches in certain fields carry more weight than others.</p> 
<p>A typical multi match query looks like this:</p> 
<div class="hide-language"> 
 <pre><code class="lang-json">{
  "query": {
    "multi_match": {
      "query": "Coral",
      "fields": [
        "title^3",
        "description",
        "color"
      ],
      "type": "best_fields"
    }
  }
}</code></pre> 
</div> 
<p>You can explore <strong>Wildcard Match</strong>, <strong>Range Filter</strong>, and other search features through the search application. For developers and administrators managing this search infrastructure, OpenSearch Dashboards is a native, developer-friendly interface for indexing, searching, and managing your data. It serves as a comprehensive control center where you can interact directly with your indices, test queries, and monitor performance in real time. The following screenshot shows OpenSearch Dashboards which provides an interactive UI to explore, analyze and visualize search and log data.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/08/BDB-5102-image-4.jpeg"><img alt="OpenSearch Dashboards" class="alignnone wp-image-79056 size-full" height="557" src="https://d2908q01vomqb2.cloudfront.net/b6692ea5df920cad691c20319a6fffd7a4a766b8/2025/06/18/BDB-5102-image-4.jpeg" style="margin: 10px 0px 10px 0px;" width="1287" /></a></p> 
<p>While our example demonstrates lexical search functionality on a sample product catalog, OpenSearch Service is equally powerful for observability usecases. When handling time-series data from logs, metrics, or traces, OpenSearch excels at real-time analytics and visualization. For instance, DevOps teams can index application logs and system telemetry data, then use date histograms and statistical aggregations to identify <a href="https://opensearch.org/blog/opensearch-as-a-siem-solution/">performance bottlenecks or security anomalies</a> as they occur. This real-time search allows IT teams to detect and respond to incidents with minimal delay. Using OpenSearch Dashboards, teams can create live operational dashboards that update automatically as new data streams in. For IoT applications monitoring thousands of sensors, this means temperature anomalies or equipment failures can trigger immediate alerts through <a href="https://docs.opensearch.org/docs/latest/observing-your-data/alerting/index/">OpenSearch’s alerting capabilities</a>. These observability workloads benefit from the same distributed architecture that powers our product search example, with the added advantage of time-series optimized indices and <a href="https://docs.opensearch.org/docs/latest/im-plugin/ism/index/">retention policies</a> for managing high-volume streaming data efficiently.</p> 
<p>Beyond search management, you can configure alerts for specific conditions, set up notification channels for operational events, and enable data discovery features. If you want to experiment with the same search queries we implemented in our application, you can launch OpenSearch Dashboards and use relevant index and search APIs from the <a href="https://docs.opensearch.org/docs/latest/dashboards/dev-tools/index-dev/">Dev Tools</a> section, which is an ideal environment for developing and testing before implementing in your production application. Because our OpenSearch Service cluster resides within a private subnet, you need to create a Secure Shell (SSH) tunnel to access the dashboard. For more information and steps to do this, refer to <a href="https://repost.aws/knowledge-center/opensearch-outside-vpc-ssh" rel="noopener noreferrer" target="_blank">How do I use an SSH tunnel to access OpenSearch Dashboards with Amazon Cognito authentication from outside a VPC?</a> in the Knowledge Center. So far, we’ve explored OpenSearch’s query domain-specific language (DSL). However, for those coming in from a traditional database background, OpenSearch also offers SQL and Piped Processing Language (PPL) functionality, making the transition smoother. You can explore more on this at <a href="https://opensearch.org/docs/latest/search-plugins/sql/index/" rel="noopener noreferrer" target="_blank">SQL and PPL</a> in the OpenSearch documentation.</p> 
<p>In this post, we introduced you to different types of keyword searches. You can also store documents as vector embeddings in OpenSearch and use it for <a href="https://aws.amazon.com/blogs/big-data/try-semantic-search-with-the-amazon-opensearch-service-vector-engine/" rel="noopener noreferrer" target="_blank">semantic search</a>, <a href="https://aws.amazon.com/blogs/big-data/hybrid-search-with-amazon-opensearch-service/" rel="noopener noreferrer" target="_blank">hybrid search</a>, <a href="https://aws.amazon.com/blogs/big-data/build-multimodal-search-with-amazon-opensearch-service/" rel="noopener noreferrer" target="_blank">multimodal search</a>, or to implement <a href="https://aws.amazon.com/blogs/big-data/supercharge-your-rag-applications-with-amazon-opensearch-service-and-aryn-docparse/" rel="noopener noreferrer" target="_blank">Retrieval Augmented Generation (RAG) pattern</a>.</p>
